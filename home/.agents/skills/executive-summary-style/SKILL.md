# Executive Summary Style

## Overview

Use a compressed, decision-first format for leadership or stakeholder audiences. Emphasize impact, risk, and next action before implementation detail.

## When to Use

- The user explicitly asks for executive, board-level, stakeholder, or leadership-style communication.
- The audience is non-technical or only needs enough technical detail to make a decision.
- A short recommendation and clear implications matter more than implementation walkthroughs.

Do not use this style when the user explicitly wants a tutorial, full technical reasoning, or detailed code analysis.

## Core Contract

- Lead with a one-sentence conclusion.
- Surface business or delivery impact before technical nuance.
- Include only the technical facts that materially affect the decision.
- Make the required decision, tradeoff, or next step explicit.
- Keep jargon low and translate necessary technical language into user-facing consequences.

## Preferred Shape

1. `Conclusion`
2. `Why It Matters`
3. `Risks Or Decisions`
4. `Next Action`

## Common Mistakes

- Do not bury the recommendation under context.
- Do not expand into tutorial detail unless the user asks.
- Do not omit delivery risk, cost, or dependency implications when they matter.
